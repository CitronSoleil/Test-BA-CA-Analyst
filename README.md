# Задача 1. Сделайте модель в нотации BPMN 2.0 по описанию рабочего потока

Модель процесса выдачи оборудования в нотации BPMN 2.0. Ниже представлен список выявленных противоречий и вопросов к описанию."![Схема процесса](file:///C:/Users/Elena/Downloads/diagram.svg)"

## Список вопросов к рабочему потоку:

**1. Вопросы к роли «Пользователь»:**
1. Пользователь делает заявку в системе No1 или всё же по телефону / почте?
2. У всех пользователей одинаковый уровень доступа ко всему оборудованию? Есть ли лимит по стоимости оборудования для разных категорий сотрудников?
3. Оценка пользователя о работе техподдержки в какой системе выставляется? По какой шкале и на что она влияет?
4. Видит ли оценку пользователя снабжение или поставщик? Ведь если задержка произойдёт на их стороне, а техподдержка не может получить низкую оценку, но в целом качество работы будет неудовлетворительным.

**2. Вопросы к роли «Техническая поддержка (1-я линия)»:**
1. Зачем конкретно нужна система No2, если в ней только дублируется заявка пользователя? Не проще ли оставить только систему No1 и не делать двойную работу по проставлению статусов заказов, закрытию заявок и других действий?
2. Если первая линия техподдержки делает заявку в снабжение в системе No2, то в почте она дублируется? Уведомления могут приходить при закрытом ЛК системы No2, поп-ап, например, и если отдел снабжения не проверил почту? Чтобы не пропустить срочную заявку.
3. Как техподдержка отражает приемку оборудования между получением от отдела снабжения и передачей пользователю? В какой системе это происходит?
4. Исходя из анализа системы, складывается впечатление, что техподдержка служит просто системой передачи заявок и оборудования. Есть ли возможность соединить пользователя с отделом снабжения напрямую? Т.о., чтобы пользователь отправлял заявку прямо в систему No3. Снабжение видело поступившую заявку (поп-ап и счетчик уведомлений в иконке изображения на панели задач, например), и сам уже напрямую связывался с поставщиком.

**3. Вопросы к роли «Снабжение»:**
1. Отдел снабжения или отдел логистики всё же отправляет оборудование в техподдержку?
2. Каким способом отдел снабжения получает подтверждение от поставщика о поступлении оборудования? В какой-то системе это регистрируется?
3. Снабжение проверяет наличие оборудования, а техподдержка сама видит остатки? Если бы видели, заявка могла бы не уходить в снабжение зря, если склада нет.
4. Сколько снабжение может ждать поставку оборудования? Сколько времени есть у логиста на доставку? Такие процессы как-то регламентируются?
5. Кто именно в отделе снабжения отвечает за контроль остатков, а кто за работу с поставщиками? Принимает заявку от техподдержки и выдает товар один и тот же сотрудник? Или есть ещё какие-то внутренние шаги, не отражённые в этой схеме?
6. Что происходит, если на шаге «Снабжение проверяет наличие» выясняется, что оборудования нет ни на складе, ни у поставщика (снято с производства)? Или если поставщик привез брак?
7. Как снабжение в системе №2 понимает, к какой заявке из системы №1 относится оборудование? Есть ли сквозной номер, или они копируют ФИО пользователя вручную? В этом случае есть риск перепутать заказы.
8. Когда оборудование выдано, кто вносит изменения в систему №3, что этот конкретный ноутбук теперь числится за пользователем А.?

**4. Вопросы к роли «Логист»:**
1. Логист - это отдел в компании или аутсорс? Каким образом вызывается логист, в какой системе он отражается?
2. Если пользователь и техподдержка в одном здании, зачем вызывать логиста? Нужно уточнить, в каких случаях включается этот шаг, например, только для региональных офисов.

**5. Общие вопросы к рабочему потоку:**
1. В процессе задействовано много ручных коммуникаций через почту (от пользователя к ТП, от снабжения к логисту и поставщику). Как минимизировать риск потери писем в спаме или их игнорирования из-за перегрузки сотрудников?
2. Требуется ли подтверждение оплаты счета перед тем, как поставщик привезет товар? Если да, то как это встроено в процесс, чтобы не увеличивать срок ожидания оборудования?

# Задача 2. Сформулируйте требования для реализации новой фичи (функциональности) на маркетплейсе

## User Story

Как **продавец маркетплейса**,  
Я хочу иметь возможность **самостоятельно загружать данные** о товаре и **отправлять их** на публикацию,  
Чтобы мои **товары стали доступны** покупателям на витрине и я мог начать продажи.  
**Приоритет** – Высокий  
**Критерии приёмки:**  
1. Реализована форма создания товара с полями: Название, Описание, Цена, Категория, Фото.
2. Реализована проверка обязательных полей перед отправкой.
3. Данные успешно сохраняются в БД со статусом «На модерации».
4. Пользователь получает визуальное подтверждение об успешном действии.

## Use Case (Загрузка товаров вручную) 

**1. Успешный сценарий** 
1. Продавец нажимает кнопку добавления товара / создания карточки.
2. Система предлагает выбрать категорию.
3. Продавец выбирает нужную категорию товаров («другое», если нужной категории нет)
4. Система подгружает список обязательных полей для выбранной категории: название, описание, цена, фото.
5. Продавец заполняет данные и загружает фото.
6. Продавец нажимает кнопку отправки.
7. Система проверяет валидность данных.
8. Система отправляет товар на модерацию.
9. Система выдает сообщение об успехе.

**2. Альтернативный сценарий 5-1. Таймаут**

1. При отсутствии ответа в течение n секунд, система сохраняет последнее заполненное поле.
2. Продавец возвращается после перезагрузки и видит заполненные поля до возникновения проблемы.
3. Возврат к шагу №5.

**3. Альтернативный сценарий 5-2. Сохранение черновика**

1. Продавец нажимает кнопку "Сохранить" / "Записать", не отправляя при этом карточку на модерацию.
2. Система сохраняет введённые значения.
3. Система отправляет уведомление, о том, что данные будут храниться n времени (24 часа, 5 дней и т.п.)

**4. Альтернативный сценарий 7-1. Ошибка валидации**

1. Система обнаруживает невалидные данные, например, цена = 0 или неверный формат данных.
2. Система подсвечивает ошибки и блокирует кнопку отправки.
3. Возврат к шагу №5.

**5. Альтернативный сценарий 7-2. Ошибка загрузки фото**

1. Система обнаруживает неподходящий формат или размер фото.
2. Система выводит подсказку, какие файлы не подходят требованиям.
3. Возврат к шагу №5.

# Задача 3. Опишите REST API интерфейса и пошаговый алгоритм создания Пользователя

## Описание REST API

**Method:** POST  
**URL:** https://demoqa.com  

**Входные параметры**
|Наименование|Тип|Обязательность|Ограничения|
|-|-|-|-|
|FirstName|String|Да|Мин. 1 символ|
|LastName|String|Да|Мин. 1 символ|
|UserName|String|Да|Уникальное|
|Password|String|Да|Мин. 8 символов,  мин. 1 спецсимвол,  одна цифра 0-9,  мин. 1 заглавная буква A-Z,  мин. 1 строчная буква a-z|
|CapchaToken|String|Да|Токен, полученный от Google reCAPTCHA после клика|

**Выходные параметры при ответе 201 - Created**  
|Наименование|Тип|Обязательность|Ограничения|
|-|-|-|-|
|userId|String/UUID|Да|Уникальный ID созданного пользователя|
|message|String|Да|Текст подтверждения регистрации|

**Выходные параметры при ответах 4ХХ-5ХХ - Ошибки**  
|Наименование|Тип|Обязательность|Ограничения|
|-|-|-|-|
|errorCode|String|Да|Технический код ошибки|
|errorMessage|String|Да|Текст ошибки для отображения пользователю|

**Описание ошибок** 
|HTTP код|Тип|Текст сообщения|
|-|-|-|
|400 Bad Request|Клиентская|"Passwords must have at least one non alphanumeric character, one digit ('0'-'9'), one uppercase ('A'-'Z'), one lowercase ('a'-'z'), one special character and Password must be eight characters or longer."|
|400 Bad Request|Клиентская|"Please verify reCaptcha to register!"|
|406 Not Acceptable|Клиентская|"User exists!"|

**Пример запроса**  
```
json{  
  "FirstName": "ivan",  
  "LastName": "ivanov",  
  "UserName": "ivan",  
  "Password": "Aa1!2zZ?",  
  "CapchaToken": "v4_token_example_12345abcdef"  
}
```
**Пример ответа для успешного сценария**  
```
json{  
  "userId": "000a0000-a00a-11a0-b111-10101010101010",  
  "message": "User Register Successfully"  
}
```
**Пример ответа для сценария с ошибкой 406**  
```
json{  
  "errorCode": "406 Not Acceptable",  
  "errorMessage": "User exists!"  
}
```

## Алгоритм создания Пользователя на стороне бэк сервиса

После нажатия кнопки «Register» бэк-сервис выполняет следующие шаги:
1. Получение JSON-тела запроса.  
2. Проверка структуры JSON (валидность синтаксиса).
3. Сервер извлекает ```CaptchaToken``` и отправляет запрос к API Google reCAPTCHA. Если токен невалиден или отсутствует — возврат ошибки **400 Bad Request** ("Please verify reCaptcha...").
4. Проверка наличия обязательных полей (```FirstName```, ```LastName```, ```UserName```, ```Password```).
5. Проверка соответствия Password политике безопасности (длина, спецсимволы, регистр). При нарушении — возврат ошибки **400 Bad Request** с описанием причины.
6. Поиск в БД записи с аналогичным ```UserName```. Если имя занято — возврат ошибки **406 Not Acceptable** ("User exists!").
7. Хеширование пароля (использование алгоритма Argon2 или BCrypt).
8. Генерация уникального ```UserId``` в формате UUID v4.
9. Выполнение транзакции: запись данных пользователя в таблицу ```Users``` (со статусом «Активен» или «Требует подтверждения»).
10. При успехе — возврат статуса **201 Created** с ID пользователя и сообщением об успешной регистрации.
